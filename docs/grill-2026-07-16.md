## A. Owner rulings tonight (pending confirmation)

**R1 — Durability boundary: Pi-session crash only.**
Precise definition (advisor round 2): V1 guarantees replacement of a failed or exhausted Pi
*model session* while the controller host process remains alive. Controller-process exit,
terminal-induced process exit, machine reboot, and machine loss are unsupported failure
boundaries in V1. "AFK" therefore means *unattended live-process operation* — pre-authorized
transitions only, persisted pending-authority state when authority is missing, best-effort
notifications while the host lives — not background-service durability. Existing ledger
files may remain after a reboot, but reboot recovery is not supported or tested in V1.
Tension to resolve on the surface: SPEC user stories 4/11 (resumable across resets, portable
handoff, SPEC.md:305,312) are stronger promises — cross-machine handoff is explicitly
post-V1, not merely unpolished wording.

**R2 — Pilot epistemology: one real delivery + controlled drills.**
Plan 020 exercises judgment→authority→implementation→independent verification→acceptance→
explicit PR publication naturally. Coordinator model-session replacement (while the
controller host remains alive), fresh-context resume, AFK pause, stale revision, duplicate
command, and cleanup are exercised as deliberate drills around the run.
Deliberation/replanning only if a real material question emerges; otherwise a separate
bounded fixture. This amends SPEC.md:222-233 ("first run must exercise" list).

**R3 — Autonomy, partially ruled: non-material graph revision is autonomous (Example 1).**
The coordinator may revise the execution graph inside the approved envelope without waking
the owner, with every revision versioned and journaled. NOT yet ruled (residual, on the
surface as its own Review Unit): ratification semantics (Example 2), knowledge-promotion
authority (Example 4), and uncertain-materiality behavior (Example 5 — pause vs
proceed-flagged). Recommendation stands: controller-validated ratification; all promotion
owner-approved in the pilot; pause when unsure.

**R4 — V1 is a concise, simple loop, improved incrementally; orchestration quality is the focus.**
The whole harness is planned, but delivered in increments. The product is a custom harness,
customizable per repository, providing a self-improving user and agent interface. Per-repo
objectives are a first-class requirement, not a later profile feature:
- **embabel-me**: pre-production, highly sophisticated — needs heavy upfront design thinking
  and judgment (front-loaded dossier, deliberation, architecture rigor).
- **PhotoQuest**: consumer production, wedding-critical at points in time — needs easy,
  fast review judgment at the end (evidence-dense review surface, browser proof, low ceremony).
This concretizes LEDGER "Repository Policy Examples" (LEDGER.md:64-78) into V1 scope,
and reframes deferred item "multiple workflow profiles" — V1 needs at least TWO repository
packages with different quality envelopes over ONE shared loop.

### RU-1 · The V1 loop shape (new — forced by R4)

**Tension.** SPEC's model-authored execution graph + controller validation
(SPEC.md:216-221) is the end-state; R4 demands a concise simple V1. What is the smallest
loop that is honest about the end-state architecture?

**Verified facts.** Ch03: an orchestration rule verbatim in context never fired (0/654
entries); shipped fine hand-chained. Ch04: making the dispatch decision explicit changed
the record, not the behavior — and demanded the decision before its deciding evidence
existed. Two chapters, two legitimate fallbacks, zero workflows — nothing has exercised a
dynamic-graph default yet. Quality-loop ledger separates durable invariants from
run-specific choices (current-quality-loop.md:17-35).

**Routes.**
- (a) **Fixed minimal loop, graph-shaped ledger** *(recommended)*: V1 hardcodes one
  sequence — intake → scout → judgment snapshot → owner gate → implement (single writer)
  → independent verify → owner acceptance → publish → analyze → cleanup — but RECORDS it
  as a typed graph in the ledger, so the graph vocabulary is real from day one and the
  coordinator earns graph-mutation authority incrementally (first: skip/insert optional
  nodes = the already-ruled non-material revision; later: full proposal).
- (b) Full model-authored graph in V1 — matches SPEC now, contradicts R4 and the empirical
  evidence that the dynamic path is unexercised.
- (c) Pure hand-chained script, no graph vocabulary — simplest, but V2 then rewrites the
  ledger schema; the terminal/event contract wouldn't survive.

**Recommendation:** (a). The evidence says the fixed path ships and the dynamic path is
unproven; R4 says make the loop excellent first. Graph-shaped records keep the end-state
honest without betting V1 on model-authored planning.

### RU-2 · Ledger placement and V1 storage (advisor branch B, narrowed by R1)

**Routes.** (amended after advisor round 2: a run directory under the repository root IS
inside the pilot's isolated worktree — my original route (a) collapsed into (c); the anchor
must be worktree-independent.)
- (a) **Repository-associated, worktree-independent anchor** *(recommended)*: run state
  keyed to the repository but outside any disposable worktree — under the git *common*
  directory area or a designated primary-checkout state directory shared across worktrees
  (e.g. `<main-checkout>/.pi-workbench/runs/<run-id>/`, git-ignored). Snapshots as JSON
  files + append-only JSONL event tail; artifacts content-addressed in a sibling `objects/`
  dir. Inspectable, no daemon, matches Ralph's proven repo-local state
  (photoquest-ralph.md:41-48) while surviving worktree cleanup.
- (b) User-scoped app storage (`~/.pi-workbench/<repo-id>/`) — cleanest worktree
  independence, weaker "cd into the repo and look at the run" ergonomics.
- (c) Inside the isolated worktree — dies with worktree cleanup; wrong.

**Facts.** SPEC.md:116 already fixes snapshots + bounded event tail + content-addressed
artifacts. Advisor: bounded *stream* tail must not be the audit log — semantic records
(decisions, approvals, graph revisions, deliberation, publication receipts) must persist
beyond it. Adopt that split in V1: `events.jsonl` may rotate; `decisions/`, `approvals/`,
`graph/`, `dossier/` never rotate silently.

### RU-3 · Minimum command/event contract (deferred #4, sized for V1)

**Recommendation.** Commands are idempotent intents: `{commandId, actor, expectedRunRevision,
verb, refs}`. Verbs for V1: `status`, `approve`, `reject`, `respond`, `pause`, `resume`,
`stop`, `publish`, `promote`, `cleanup`. Events are immutable facts: `{runId, seq, schemaVer,
actor, type, causationId, artifactRefs}`. Duplicate `commandId` returns the recorded result
(the drill in R2 tests exactly this). Amended after advisor round 2: RU-1 + R3 grant the
coordinator narrow autonomous graph mutations, so V1 MUST also include one internal,
controller-mediated `graph-mutate` transition — restricted to contract-enumerated optional
nodes, carrying prior graph revision, mutation class, rationale, evidence refs, and the
resulting revision; the controller accepts or rejects it. Terminal owner-steering verbs and
arbitrary model-authored graph proposals stay V2. Without this transition the "graph-shaped
ledger" is just a visualization of a script.

### RU-4 · Mechanical invariants vs judgment attestations (advisor branch D — his #1 assumption attack)

**Tension.** SPEC says the controller "validates graph mutations against quality, authority,
safety, budget, isolation, independence, and evidence requirements" (SPEC.md:352). Only some
of those are machine-checkable. Blurring this makes the controller a rubber stamp or a hidden
orchestrator.

**Recommendation.** Split every envelope obligation into: (i) mechanical invariants —
cycles, budget, permissions, lease non-overlap, required fields, approval references,
author≠reviewer identity — controller-enforced, fail-closed; (ii) judgment attestations —
evidence sufficiency, materiality, independence-in-substance, dossier completeness — carried
as typed, provenance-bearing attestations by a named model or the owner, challengeable and
never marketed as deterministic validation. Ch03's laundered scout claim (six findings) is
the standing example: durability preserved the error, not its epistemic status. V1
consequence: claims carry `verified|unverified` marks that synthesis must transform, not
retype — ch04 lost 2 marks in synthesis (R3b PARTIAL).

### RU-5 · Autonomy residuals (deferred #1 and #7, after R3)

Ratification (Ex.2): controller-validated synthesizer action, human approval reserved for
material direction, authority expansion, acceptance, publication *(recommended)* vs
human-ratifies-every-snapshot. Promotion (Ex.4): all owner-approved in pilot *(recommended)*
vs auto-promote narrow mechanical classes. Uncertainty (Ex.5): pause *(recommended)* vs
proceed-flagged. Advisor and I align on all three; the owner picks tomorrow.

**Materiality bound (advisor round 2, connects RU-1/4/6):** "non-material" must be
mechanically bounded, not model-judged — only contract/package-named optional nodes may be
skipped or inserted; invariant gates (pre-mutation authority, independent verification,
final acceptance, explicit publication, analysis, cleanup) cannot be removed;
package-required evidence cannot be weakened; every other topology change is material or
uncertain. Otherwise the coordinator can autonomously relabel policy weakening as
"non-material".

### RU-6 · Per-repo workflow packaging (forced by R4 — supersedes "one contract only")

**Tension.** SPEC.md:201 says the initial implementation "does not select, compose, inherit,
or resolve multiple workflow contracts", and LEDGER #19 defers profiles. R4 makes two repos
with opposite quality centers a V1 requirement.

**Routes.** (amended after advisor round 2: "gate strictness and evidence class per node"
is already the beginning of a profile DSL — the honest boundary is emphasis, not topology.)
- (a) **One fixed stage topology + two repository packages with finite contract-owned
  policy fields** *(recommended)*: both repos run the same ordered semantic stages; the
  package is statically selected by the target repository and may vary only *depth and
  evidence* within contract-defined fields — embabel-me: deep pre-implementation dossier,
  stronger options/architecture evidence, deliberate challenge where real uncertainty
  exists, *concise but still independent* post-implementation reconciliation (NOT "relaxed
  end review" — invariant gates cannot be disabled); PhotoQuest: concise pre-mutation
  revalidation, mandatory browser/Playwright evidence, evidence-dense fast end review
  (LEDGER.md:66-78). No composition, no inheritance, no personal/run overlays, no topology
  mutation by package. The resolved package + envelope are hashed and persisted at run
  creation; a replacement coordinator resumes from the recorded resolution — a changed
  package triggers reconciliation, never silent adoption.
- (b) Two workflow contracts — full deferred-profiles machinery early; contradicts R4.
- (c) One contract, PhotoQuest only; embabel-me waits — loses the design-heavy validation
  case entirely and lets the loop overfit to review-heavy delivery.

**Open question the owner must answer:** does R4 mean *different emphasis within one
topology* (recommended) or *genuinely different sequences/mandatory stages*? If the latter,
RU-6 is hiding two workflow profiles and RU-1's single fixed loop is false.

**Validation obligation:** the pilot exercises only PhotoQuest, so V1 needs one bounded
embabel-me exercise — resolve its package, run intake + deep pre-implementation judgment on
a real design question, produce and ratify the dossier, stopping before mutation is fine.
Without it, embabel-me is a configuration example, not validated V1 behavior.

**PhotoQuest "easy end review" in V1 terms:** terminal/artifact interpretation only —
concise acceptance summary, criteria mapped to evidence, browser artifacts referenced
directly, deviations/residual risks isolated, one explicit acceptance action. It must not
silently depend on the future generated-surface UI.

### RU-7 · Model binding (deferred #5)

**Recommendation** (advisor concurs; SPEC.md:337 already hybrid): workflows declare role
capability/effort/continuity/independence; repository policy resolves or pins the model;
every dispatch records the exact binding; no silent capability-weakening fallback — a
fallback either satisfies a policy-approved equivalence or raises an authority request.
Routing quality is measured per role from run analysis (model-routing-x.md:34-38), not fixed
from social heuristics.

### RU-8 · Analysis cadence + retention defaults (deferred #2, #3, sized for V1)

Per-run analysis mandatory (settled). Cross-run: **owner-started command only in V1**
*(recommended — R4)*; thresholds later; no periodic daemon. Retention V1 defaults: keep
envelope, approvals, decisions, graph revisions, dossier snapshots, claim/deliberation
records, evidence manifests, publication receipts, analysis, candidate dispositions; expire
raw transcripts, verbose tool output, rebuildable caches, scratch — short windows unless
pinned. Ralph's unbounded attempt history is the cautionary case (photoquest-ralph.md:52).

### RU-9 · Failure, orphans, corrections (advisor branches C/L, simplified by R1)

Under R1 the orphan matrix shrinks: coordinator session dies → new session resumes from
ledger (the drill proves it). Still needed in V1: an explicit `unknown-outcome/reconcile`
state for a worker or external write whose result was lost (binary success/failure is
insufficient — advisor C); append-only corrections — authoritative records are superseded
or tombstoned with a correction event, never edited (the ch04 false-attribution incident is
the motivating case). Machine-reboot and machine-loss recovery: explicitly out of V1 scope,
recorded as such.

### RU-10 · Workspace integrity (advisor branch F)

Leases are advisory in V1 + **mutation detection**: revision/dirty-state check before
mutation and before commit; unknown external change → reconciliation event with
`actor=unknown`, pause. No enforced locks in V1. Evidence: live observation showed lease
records don't create exclusivity and misattribution happens when the run can only see its
own agents.

### RU-11 · Skills track: overlay representation now, surfaces later (deferred #8-11)

**Overlay representation (deferred #11)** — has a working prototype:
`vendored/MANIFEST.json:4-9` records source, skillPath, `vendoredFromHash`,
`upstreamUpdatedAt` for 18 skills. *(Recommended)*: vendored base immutable +
provenance-bearing; stack/repo/run adaptations as separate overlay layers keyed to base
content identity; conflicts fail closed; promotion = new reviewed layer version, never an
in-place rewrite (this ratifies the LEDGER resume brief's pending recommendation,
LEDGER.md:118). Advisor caveat adopted: `vendoredFromHash` identifies folder content, not
an immutable upstream commit — add the upstream commit/release id to the manifest.
**Interface scope / generation moment / surface promotion (deferred #8-10)** — *(recommended)*
defer past V1 except the headless/terminal path; when they land: small stable API
(progress, typed I/O, decisions, authority requests, actions, artifact refs, errors,
cancellation); generate on first meaningful use, not on add (the "when I add a skill" user
story SPEC.md:327 is aspirational); promotion via typed candidate + human-reviewed package
change.

### RU-12 · Pilot drill list + falsification criteria (advisor branch J, required by R2)

Drills (amended after advisor round 2 — every drill scoped to R1's supported boundary):
terminate/invalidate the coordinator *model session* while the controller host remains
alive → attach a fresh session and resume solely from persisted run state; duplicate
`publish` command delivery → single PR (idempotency within the live-controller boundary —
a controller-process death after the PR call but before receipt persistence is outside R1);
stale-revision approval → rejected; AFK missing-authority → pending-authority state
persisted *before* best-effort notification, no further unauthorized transition; unknown
external workspace mutation → detected, `actor=unknown` reconciliation event; cleanup
preview → pinned evidence survives. Explicitly NOT V1 acceptance criteria:
controller-process crash recovery, terminal-exit recovery, machine-reboot recovery, worker
continuation under controller death, cross-machine handoff.
**Falsifiers** (any one observed = architecture defect, not incident): action without
durable authority record; resume requires conversation history; duplicate external
publication; silent continuation over unknown external mutation; verified/unverified mark
lost through synthesis; cleanup deletes referenced evidence.

### RU-13 · Secrets and artifact confidentiality (advisor branch H — V1-scoped)

Content-addressed artifact stores retain whatever workers saw. V1 minimum (advisor round 2:
present as a scope acknowledgment, not a design branch): run state local-user-readable only;
run directory git-ignored; explicit `do-not-persist` artifact classes; workers must not
intentionally persist credentials or production personal data; pattern-based redaction is
best-effort hygiene, NOT a security boundary and never claimed as comprehensive secret
detection. Access control, encryption, multi-user isolation deferred.

### RU-14 · V1 slice of deliberation and compounding (new — advisor round 2 gaps 5/6)

**Tension.** The roadmap deferred deliberation threads to V2 and compounding to V3, but
R4 makes embabel-me's heavy upfront judgment V1-first-class (needs durable challenge/
response), RU-4 requires claim records whose status survives synthesis, RU-3 includes
`promote`, and RU-5 asks the owner to decide pilot promotion authority. Deferring both
contradicts four other units.

**Recommendation.** V1 includes the *minimal* durable form of each: (i) typed
challenge/response/claim-status records with synthesis references — no rich multi-party
thread UX, and no run is forced to deliberate; (ii) run analysis may produce coordinator-
authored typed learning candidates, and the owner promotes or rejects every candidate
before cleanup — no dedicated compounder, no automatic extraction, no cross-run machinery.
Rich threads = V2; automatic compounding/auto-promotion = V3.

## D. Incremental roadmap (amended after advisor round 2)

**V1:** fixed semantic loop represented as a typed graph; narrow allowlisted graph
mutations via an internal `graph-mutate` transition; one statically resolved repository
package per repo (embabel-me + PhotoQuest), package/envelope hashed and persisted; terminal
command/event contract; minimal durable claims + challenge/response records; pre/post
judgment artifacts; per-run analysis; typed learning candidates with owner disposition;
session-replacement and integrity drills; one bounded embabel-me judgment-path exercise.
**V2:** arbitrary coordinator graph proposals inside the envelope; richer deliberation
thread mechanics and UX; broader steering commands; stronger controller/process isolation
if desired; richer event projections.
**V3:** dedicated compounder; cross-run analysis; automatic candidate extraction; evaluated
narrow auto-promotion; generated skill surfaces and surface promotion.
